# Explanation review criteria

## What an explanation must do

An explanation **deepens understanding**. The reader isn't trying to do anything right now — they want to understand *why* something works the way it does, or *how* a concept fits together. Success is insight, not action.

## Structure checklist

- [ ] Title is a question ("Why does X work this way?") or a concept name ("How caching works")
- [ ] Opens by stating what the reader will understand by the end
- [ ] Builds understanding progressively — each section prepares for the next
- [ ] Uses analogies, examples, and diagrams where they aid understanding
- [ ] Ends with a summary of the key insight and pointers to related how-to guides or reference docs

## Content checklist

- [ ] Explains *why*, not just *what*
- [ ] Doesn't include task instructions — those go in how-to guides
- [ ] Doesn't duplicate reference data — link to the reference doc instead
- [ ] Handles alternative viewpoints and tradeoffs where relevant ("You might wonder why X rather than Y — the reason is...")
- [ ] Uses concrete examples to anchor abstract concepts
- [ ] Length is proportionate to the complexity of the concept — don't pad, don't truncate

## Common explanation doc failures

**No "why"**: Describing what something does without explaining why it was designed that way or what problem it solves. That's reference content.

**Hidden how-tos**: An explanation that includes "to configure this, follow these steps." Extract the steps into a how-to guide and link to it.

**Too abstract**: A concept explained only in abstract terms, with no examples or analogies. If a reader can't picture it, they don't understand it.

**Trying to be everything**: An explanation that also includes a tutorial and a reference table. Split it.

**No payoff**: The reader reaches the end and isn't sure what they're supposed to understand differently. The closing summary should make the key insight explicit.
